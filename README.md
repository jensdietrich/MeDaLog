# MeDaLog

MeDaLog is a set of _annotation processors_ for standard [Souffle](https://souffle-lang.github.io/) Datalog programs.
This is inspired by 
[annotation processing in Java](https://jcp.org/en/jsr/detail?id=269), 
[go generate](https://go.dev/blog/generate), 
[typescript generators](https://www.typescriptlang.org/docs/handbook/decorators.html)
and similar constructs in other languages. 


## Overview

An annotation processor transforms valid Souffle programs into valid Souffle programs.

MeDaLog provides two such annotation processors:

- **Identity processor** (`-p id`) – adds an `id: symbol` slot to every declared predicate,
  assigns provenance IDs to facts and rules, and rewrites rule heads to encode proof structure
  using Souffle's `cat` function. This can be used to provide an eager provenance for Souffle programs, 
  where each inferred fact encodes the inference used to compute it. 
- **Metadata processor** (`-p metadata`) – processes `@[key = value]` metadata annotations on
  components and rules and generates a `.comp _metadata` component containing `annotation` facts and
  inheritance rules. `annotation` are made part of the generated database, and can be queried without
  having to parse the annotations. Rules can inherit annotations from components.

Both processors take a standard Souffle program as input and produce a transformed Souffle program
as output. The input uses native Souffle annotation syntax (`@[key = "value"]`) and Souffle
component blocks (`.comp Name { ... }`). [Souffle](https://souffle-lang.github.io/) ≥ 2.5 supports those language features.

## Requirements

- Java 17
- Maven 3.8+
- [Souffle](https://souffle-lang.github.io/) ≥ 2.5 (optional for build; required only to run the generated
  programs and to execute Souffle-validation tests)

## Getting the classpath

```bash
mvn dependency:build-classpath -Dmdep.outputFile=cp.txt
```

The classpath will be written to `cp.txt`. Use it as `<classpath>` below.

## `io.github.bineq.medalog.Main`

Runs one of the MeDaLog annotation processors on a Souffle input file and writes the transformed
program to an output file.

**Build:**

```bash
mvn package
```

This produces `target/medalog.jar` (fat jar with all dependencies bundled).

**Run:**

```bash
java -jar target/medalog.jar -p <id|metadata|all> -i <input> -o <output> [-m <keys>] [-a <fqcn>]
```

Or without the fat jar:

```bash
java -cp <classpath> io.github.bineq.medalog.Main -p <id|metadata|all> -i <input> -o <output> [-m <keys>] [-a <fqcn>]
```

**Options** (generated by commons-cli):

```
usage: medalog -i <arg> [-a <arg>] [-m <arg>] -o <arg> -p <arg>
 -a,--aggregation <arg> Fully-qualified class name of an AggregationStrategy
                        implementation (default: CatAggregationStrategy)
 -i,--input <arg>       Input Souffle program file
 -m,--metadata <arg>    Comma-separated metadata annotation keys (e.g.
                        author,description,created)
 -o,--output <arg>      Output Souffle program file
 -p,--processor <arg>   Annotation processor to run: id | metadata | all
```

**Examples:**

```bash
# Add provenance IDs to all predicates (default cat-based aggregation)
java -jar target/medalog.jar -p id -i program.dl -o program-id.dl

# Add provenance IDs using a custom aggregation strategy
java -jar target/medalog.jar -p id -a com.example.MyStrategy -i program.dl -o program-id.dl

# Extract metadata annotations into a _metadata component
java -jar target/medalog.jar -p metadata -m author,description -i program.dl -o program-meta.dl

# Run both processors (identity first, then metadata)
java -jar target/medalog.jar -p all -m author,description -i program.dl -o program-out.dl
```

## Annotation syntax

MeDaLog uses the native Souffle outer annotation syntax. Annotations are placed before a
component or rule:

```souffle
@[author = "jens"]
@[description = "family rules"]
.comp Family {
    .decl parent(p: symbol, c: symbol)
    .decl grandparent(gp: symbol, gc: symbol)

    parent("Tim", "Bob").
    parent("Bob", "Ann").

    @[id = "gp-rule"]
    @[description = "grandparent derivation"]
    grandparent(x, z) :- parent(x, y), parent(y, z).

    .output grandparent
}

.init f = Family
```

The `id` key is reserved for the identity processor. All other keys are treated as metadata
when passed to the metadata processor via `-m`.

### Annotation inheritance

Metadata annotations are inherited by nested components and by rules from their enclosing
component. If a rule or inner component defines the same key, its value takes precedence
(annotation overriding). Inheritance is one-hop from directly asserted values.

### Identity processor output

The identity processor injects an `id: symbol` first slot into every `.decl`, prepends a
provenance ID to every fact, and rewrites rule heads to aggregate IDs from body atoms using
the configured aggregation strategy (see [ID aggregation strategy](#id-aggregation-strategy) below).

With the default `CatAggregationStrategy` the output looks like:

```souffle
.decl grandparent(id: symbol, gp: symbol, gc: symbol)
grandparent(cat("gp-rule", cat("[", cat(_id1, cat(",", cat(_id2, "]"))))), x, z) :-
    parent(_id1, x, y), parent(_id2, y, z).
```

### ID aggregation strategy

When the identity processor rewrites a rule head, the derived-fact ID is assembled from the
rule's own ID and the IDs of the body atoms. The algorithm for this assembly is encapsulated
in the `io.github.bineq.medalog.id.AggregationStrategy` interface:

```java
public interface AggregationStrategy {
    String generateId(String ruleId, List<String> bodyIds);
}
```

**Default: `CatAggregationStrategy`**

The built-in implementation uses Souffle's `cat` function to produce IDs of the form
`ruleId[id1,id2,...,idN]`, which conform to the proof-term grammar used by
[DaLeQ](https://github.com/binaryeq/daleq):

| Body atoms | Generated expression |
|------------|----------------------|
| none | `ruleId` |
| one | `cat(ruleId, cat("[", cat(_id1, "]")))` |
| two | `cat(ruleId, cat("[", cat(_id1, cat(",", cat(_id2, "]")))))` |

**Custom strategy**

Any class on the classpath that implements `AggregationStrategy` and has a no-argument
constructor can be used instead. Pass its fully-qualified class name via `-a`:

```bash
java -jar target/medalog.jar -p id -a com.example.MyStrategy -i program.dl -o program-id.dl
```

The class is loaded at runtime via reflection. An `IllegalArgumentException` is thrown if the
class is not found or does not implement `AggregationStrategy`.

### Metadata processor output

The metadata processor produces a `.comp _metadata` block containing:

```souffle
.comp _metadata {
    .decl annotation(id: symbol, annotation_name: symbol, annotation_value: symbol)
    .decl _assertedAnnotation(id: symbol, annotation_name: symbol, annotation_value: symbol)
    .decl _annotationKeyAsserted(id: symbol, annotation_name: symbol)
    .decl _compHierarchy(childId: symbol, parentId: symbol)
    .decl _member(ruleId: symbol, compId: symbol)

    _member("gp-rule", "Family").
    _assertedAnnotation("Family", "author", "jens").
    _assertedAnnotation("Family", "description", "family rules").
    _assertedAnnotation("gp-rule", "description", "grandparent derivation").

    // ... inheritance rules ...

    .output annotation
}
.init metadata = _metadata
```

## Running tests

```bash
mvn test
```

Tests requiring the `souffle` binary are automatically skipped when it is not on the PATH.
Souffle-annotation tests (those exercising `@[...]` syntax) are additionally skipped when the
installed Souffle is older than 2.5.
